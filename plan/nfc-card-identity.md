# NFC カードによる本人識別 — 方式調査 (Unit NFC / ST25R3916)

点呼のドライバー本人確認に NFC カードを使う場合の方式整理。
実装 issue: [#84](https://github.com/ippoan/alc-app-s3/issues/84) (交通系 IDm 読み取りドライバ)。

ハードは全方式共通で **M5 Unit NFC (U216, ST25R3916, I2C/Port A)** 1台で足りる
(NFC-A/B/F/V 対応)。⚠️ Unit RFID (MFRC522) / RFID2 (WS1850S) は FeliCa 非対応。

## 結論サマリ

| カード | 方式 | かざすだけ | 日常の点呼タップ | 登録時の本人確認 (eKYC) |
|---|---|---|---|---|
| 交通系 IC / モバイル Suica | NFC-F, IDm | ○ | **◎ 採用** | ✗ (IDm は人を証明しない) |
| マイナンバーカード | NFC-B, APDU | △ (照合番号B 保存後は PIN 不要) | ○ 券面AP 経由なら可 (要事前登録) | **◎ 候補** (顔写真も取れる) |
| 運転免許証 (IC) | NFC-B, APDU | △ (期限のみ PIN 不要) | ✗ 本人識別は不可 / **○ 有効期限チェックは可** | **◎ 本命候補** (点呼は免許確認が業務要件) |

- **日常運用 = 交通系 IDm タップ** (#84)。
- **PIN 方式 (マイナンバー / 免許証の本人識別) は日常タップには使えない**。毎回 4 桁 PIN 入力が
  必須で、入力ミス累積で**カードがロック**する。かざすだけ運用は原理的に不可能。
- ただし**免許証の交付日・有効期限は PIN なしで読める** (MF 直下 EF `2F01`,
  `ippoan/AlcoholChecker` の `NfcReader.kt` で実装・動作済み)。
  「点呼時に免許証をかざして期限切れ自動検出」は日常運用に組み込める。
- ただし**初回登録時に 1 回だけ** PIN 方式で実在確認し、「この IDm はこの人」の紐付けを
  検証済みにする **二段構え**は将来資産として有力。

## 1. 交通系 IC / モバイル Suica (NFC-F, IDm) — 日常運用

詳細は #84。要点のみ:

- FeliCa Polling (システムコード `0x0003` = Suica/PASMO/ICOCA/PiTaPa/TOICA 共通) →
  応答先頭 8byte = **IDm**。認証・暗号不要領域なのでかざすだけで取れる。
- モバイル Suica (おサイフケータイ / Apple Pay) も Secure Element 上の本物の FeliCa で
  **IDm は端末ごとに固定**。IDm ランダム化は HCE-F 自作カードの話で Suica は該当しない。
- SESAME Touch の card 認証と同一方式 (sesame リポ `FINDINGS.md` 参照。
  BLE プロトコル上 `card_id`=IDm, `CARD_TYPE_SUICA=0x01` は表示用)。
- 参照実装: [M5Unit-NFC `NFCF/JapanTransportationICCard`](https://github.com/m5stack/M5Unit-NFC/tree/main/examples/UnitUnified/NFCF/JapanTransportationICCard)
  — `NFCLayerF::detect()` + `PICC::idmAsString()` のみで識別完了。ESP-IDF native 対応。
- 弱点: IDm は平文・リプレイ可 → 単独の本人確認にしない (顔認証と併用)。
  IDm→人の紐付けの信頼性は自己申告 → これを埋めるのが下記 PIN 方式の登録時利用。

## 2. マイナンバーカード (NFC-B, 券面事項入力補助AP) — 登録時候補

参照実装: [M5Unit-NFC `NFCB/JapanIDCard`](https://github.com/m5stack/M5Unit-NFC/tree/main/examples/UnitUnified/NFCB/JapanIDCard)
(ESP-IDF native 対応。`NFCLayerB` + `NFCBFileSystem`)。

読み取りシーケンス (例の実装):

1. NFC-B 活性化 (`cfg.mode = NFC::B`)
2. `selectByDfName(D3 92 10 00 31 00 01 01 04 08)` — 券面事項入力補助AP を選択
3. `verifyGlobal(暗証番号4桁)` — **券面事項入力補助用暗証番号**の照合
4. EF `0x0002` を `readBinary()` → 氏名等の券面情報

### 日常タップに使えない理由

- **毎回 PIN 入力が必須**。Type-B のカード識別子 (PUPI) はセッション毎にランダムで、
  Suica の IDm のような「かざすだけの固定 ID」が原理的に取れない。
- **PIN 連続ミスでカードロック** (例のソースにも警告あり)。点呼のたびに全ドライバーが
  PIN 入力する運用は事故率的に成立しない。
- 氏名・住所という実個人情報を端末で扱うことになり、取り扱い責務が重い
  (IDm は無意味な 8byte なので漏洩時の被害が限定的なのと対照的)。

### PIN 入力なしで読む経路 — 照合番号B (保険証顔認証と同方式)

「毎回 PIN 入力」を回避する正規の経路がある。**照合番号B** による券面事項確認AP (券面AP)
アクセス:

- **照合番号B = 14 桁** = 生年月日 6 桁 + カード有効期限の西暦 4 桁 + セキュリティコード 4 桁。
  **カード券面に印字されている** → カメラ撮影 + OCR で機械的に取得できる。
- 照合番号B で券面事項確認AP にアクセスすると、**暗証番号なしで顔写真を含む券面画像**
  (+ 生年月日・性別・有効期限。マイナンバーは含まれない) が読める。
- これは**マイナ保険証の顔認証付きカードリーダーが実運用している方式**
  (券面 OCR → 照合番号B → 券面AP から顔写真 → 利用者の顔と照合)。
- 照合番号B にも試行回数制限 (ロック) はあるが、保存済みの正しい値を送信するだけなので
  手入力ミスによるロックは起きない。

### 登録・運用フロー案 (点呼向け)

1. **初回登録**: 社員証 / 交通系 IDm 等で一時 ID を発行
2. **カード券面をカメラで撮影** → OCR で照合番号B (生年月日/有効期限/セキュリティコード) を
   取得し、driver に紐付けて**保存**
3. **以後の認証**: NFC タップ + 保存済み照合番号B で券面AP を読む —
   **PIN 入力なし・ロックリスクなし**で顔写真・有効期限が取れる
4. 取得した**顔写真は顔認証の登録ソース**にも使える (自撮り登録が不要になる)

CoreS3 単体ではカメラ・OCR がないため、②はスマホ/タブレット (alc-app) 側の担当。
NFC 読み取り自体は CoreS3 (`NFCLayerB` raw APDU) でもスマホでも可。

## 3. 運転免許証 IC (NFC-B) — 登録時の本命候補

M5Unit-NFC に**専用例はない** (NFCB は Detect / JapanIDCard のみ) が、
マイナンバーと同じ NFC-B + APDU (`NFCLayerB` / `NFCBFileSystem`) の枠で読める。
DF/EF 構成と APDU は警察庁「運転免許証及び運転免許証作成システム等仕様書」に準拠。

### IC チップの構造

| 読める情報 | PIN | 備考 |
|---|---|---|
| **交付日・有効期限** (MF 直下 EF `2F01`, 共通データ要素) | **不要** | AlcoholChecker で実装・動作済み |
| PIN 残り試行回数 | 不要 | VERIFY 空送信で照会 (試行を消費しない) |
| 氏名・免許証番号 (券面記載情報) | 暗証番号1 | |
| 顔写真・本籍 | 暗証番号1+2 | |

- 暗証番号は **4 桁 × 2 組**。**3 回連続ミスで IC がロック**し、以後読み取り不可 (解除は警察窓口)。
- → 本人識別 (氏名・免許証番号) はマイナンバーと同じ「日常タップ不可・登録時のみ」判定。
  ただし**有効期限だけなら PIN なし・ロックリスクなしで日常タップ可能** (次節)。

### PIN なしで読める領域 — `ippoan/AlcoholChecker` で実装済み

`AlcoholChecker/app/src/main/java/com/example/alcoholchecker/nfc/NfcReader.kt` が
Android IsoDep で以下を PIN なしで読んでいる (CoreS3 では `NFCLayerB` の raw APDU で同等可):

```
免許証判定:   ATR/hiLayerResponse prefix = 3B 88 80 01 00 00 00
SELECT MF:    00 A4 00 00
残回数照会:   00 20 00 81            → SW 63 CX (下位ニブル = 残り回数)
SELECT EF:    00 A4 02 0C 02 2F 01   → MF 直下 EF 2F01 (共通データ要素)
READ BINARY:  00 B0 00 00 11         → 17 バイト
              offset 10〜: 交付日 (BCD "20260310" 形式)
              offset 18〜: 有効期限 (BCD)
```

→ **点呼時の「免許証有効期限チェック」は、かざすだけ (PIN 入力なし・ロックリスクなし) で
実装できる**。誰の免許証かは分からない (Type-B PUPI はセッション毎ランダム、氏名は PIN1 必要)
ので本人識別は Suica IDm のまま。ただし「点呼時に免許証もかざして期限切れを自動検出」は
独立した価値として日常運用に組み込める。

### それでも本命である理由

- **点呼はもともと免許証確認が業務要件**。ドライバーが確実に携行している。
- 暗証番号1 だけで氏名+免許証番号が取れれば、`IDm → 免許証番号 → driver` の
  検証済み紐付けが作れる。免許証番号は既存の運行管理データと直結する。
- 弱点: 免許証の暗証番号は交付時設定のまま忘れられていることが多い
  (ロック 3 回制限と相まって運用リスク)。
  → 登録フローでは「読めなければ目視確認にフォールバック」を必須とする。

### マイナ免許証への収斂 (2025年3月〜)

運転免許証のマイナンバーカードへの一体化 (マイナ免許証) が始まっており、
将来的には **2 と 3 は同じ物理カード**になっていく。登録時 eKYC を実装するなら
NFC-B + APDU の共通基盤 (`NFCLayerB`/`NFCBFileSystem`) に乗せておけば両対応できる。

### 免許AP の技術仕様は警察庁通達で公開済み

**「非公開」ではなかった。** 警察庁 丁運発第145号 (令和6年7月19日) の別添
「**運転免許証及び運転免許証作成システム等仕様書 (V010)**」の**別紙3
「免許情報記録個人番号カードの仕様」**が、マイナ免許証の免許AP のファイル構成・AID・
アクセス権限・BER-TLV タグ表まで全て規定している。
PDF: https://www.npa.go.jp/laws/notification/koutuu/menkyo/menkyo20240719_145.pdf

仕様書 V010 別紙3 から抜粋 (実装に必要な要点):

- **搭載場所**: 個人番号カードの空き領域 (拡張利用領域) に警察独自カードAP を搭載。
- **AID**: RID `A0 00 00 02 31` + PIX (インスタンス-AID は PIX 先頭 `06`、以降 `00`)
  → SELECT するインスタンス-AID = `A0 00 00 02 31 06 00 00 00 00 00 00 00 00 00 00`
- **ファイル構成**:
  | EF | EF-ID | 内容 | アクセス |
  |---|---|---|---|
  | IEF01 | `0006` | 暗証番号 (PIN, 4桁) | — |
  | WEF01 | `001A` | PIN 設定有無 (1=設定/0=未設定) | **FREE (PINなしで読める)** |
  | WEF02 | `001B` | 免許情報記録 (BER-TLV, 最大5845B) | PIN (未設定時は DPIN) |
  | WEF03 | `001C` | 電子署名 | PIN (未設定時は DPIN) |
- **PIN 未設定カードの読み方**: WEF01 を FREE で読んで b1=0 (未設定) なら、
  **DPIN = `****` (ASTERISK `2A` ×4)** で VERIFY すれば WEF02 が読める。
  公式アプリの「暗証番号を設定していない場合」分岐の正体はこれ。
- **試行許容回数は 10 回**で閉塞 (解除は公安委員会)。従来免許証の 3 回より緩い。
- **WEF02 の主要タグ** (BER-TLV): `C5`=免許情報記録の有効期間の末日 (元号YYMMDD 7B) /
  `C4`=色区分 / `C6`〜`D3`=免許の条件1〜12 / `E7`=免許情報記録番号 (12B) /
  `EC`〜=免許の種類 (大型/普通/大特/…各1B) / `E9`〜`EB`=免許の年月日

さらに WEF02 の後続タグと WEF03 で以下も確認済み (別紙3 続き):

- タグ `107` = **モノクロ写真 (JPEG2000, 2000B)** — PIN 未設定なら**顔写真まで読める**
- WEF03 = 電子署名 (SHA-256 + RSA-2048, PKCS#1 v1.5)。WEF02 全体のハッシュに署名
  → **カードの真正性 (偽造検証) までオフラインで可能**
- 読み出しコマンドは SELECT FILE / VERIFY / READ BINARY の 3 つのみ (JIS X 6306、
  基本コマンドは同仕様書の別添3-1)。AlcoholChecker の `NfcReader.kt` パターン流用可。

### PIN 未設定のマイナ免許証は「Suica IDm の上位互換」

| | 従来 IC 免許証 (PIN なし) | マイナ免許証 (PIN 未設定) |
|---|---|---|
| 有効期限 | ○ (EF `2F01`) | ○ (`C5`) |
| 免許の種類・条件・色区分 | ✗ | ○ |
| 固定の識別番号 | ✗ | **○ 免許情報記録番号 (`E7`)** |
| 顔写真 | ✗ (PIN1+2 必要) | **○ (`107`)** |
| 改ざん検証 | ✗ | ○ (WEF03) |
| 氏名・生年月日 | ✗ | ✗ (券面AP 側) |

- `E7` が固定 ID → IDm と同じ「かざすだけ識別」。しかも免許に直結した番号。
- 顔写真が同時に取れる → その場で顔照合の照合元になる (IDm にない性質)。
- 電子署名で複製・偽造の検証可 (IDm は複製検証不能なのと対照的)。

## 全ドライバーをカバーする構成 (決定版)

PIN 設定者も含め、既出の部品の組み合わせで全パターンが埋まる:

| ドライバーの所持カード | 日常の識別 | 免許期限チェック |
|---|---|---|
| マイナ免許証 (PIN 未設定) | かざすだけ: DPIN → `E7` + 顔写真照合 | 同時に `C5` で取得 |
| マイナ免許証 (PIN 設定) / 通常マイナンバーカード | 初回: 一時 ID + 券面撮影 OCR → **照合番号B 保存**。以後かざすだけ: 券面AP → 顔写真 + 生年月日で識別・照合 | 免許期限は WEF02 に入れない (PIN 必要) → **アプリ (公式読み取りアプリ / スマホ側) or 台帳**で管理 |
| 従来 IC 免許証 + 交通系 IC | Suica IDm (#84) | EF `2F01` を PIN なしで取得 (AlcoholChecker 方式) |
| 交通系 IC のみ | Suica IDm (#84) | 台帳管理 |

- どの経路も**日常運用で PIN 手入力なし・ロックリスクなし**。
- 判定はかざした瞬間に機械的にできる: NFC-F なら IDm、NFC-B なら
  免許AP の WEF01 (FREE) → 未設定なら DPIN 経路 / 設定済みなら券面AP (照合番号B) 経路。

## 実装への含意

1. まず #84 (交通系 IDm) を実装。スパイク① (M5UnitUnified NFC コアのヘッドレス切り出し)
   が通れば、**NFC-B 系も同じ切り出しで動く** (NFCLayerB は同一ライブラリ)。
2. 登録時 eKYC (免許証/マイナンバー + PIN テンキー UI) は別 issue として将来起票。
   rust-alc-api 側の driver 紐付け設計と同時に決める。
3. PIN 方式を実装する場合はロック対策 (残り試行回数の警告表示、目視フォールバック) を必須要件にする。
